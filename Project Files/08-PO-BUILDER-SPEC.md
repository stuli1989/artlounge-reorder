# 08 — Purchase Order Builder Specification

## Overview

The PO Builder generates purchase orders based on computed reorder data. It pre-fills order quantities using velocity × lead time × safety buffer, allows manual adjustment, and exports as Excel or PDF.

## Order Quantity Calculation

### Formula

```
Suggested Order Qty = (wholesale_velocity + online_velocity) × lead_time_days × safety_buffer
```

### Breakdown

- **Wholesale velocity:** The primary demand driver. Units/day based on in-stock days.
- **Online velocity:** Background drain. Always included because both channels draw from the same stock pile.
- **Lead time:** Days from order placement to stock arriving in warehouse. Configurable per supplier, with option to switch between sea and air freight.
- **Safety buffer:** Multiplier to account for demand variability. Default 1.3x (30% extra). User-adjustable in the PO Builder UI.

### Example: Speedball Sealer

```
Wholesale velocity:  1.60 units/day
Online velocity:     0.20 units/day
Total velocity:      1.80 units/day

Sea freight (180 days):
  Suggested = 1.80 × 180 × 1.3 = 421 units

Air freight (30 days):
  Suggested = 1.80 × 30 × 1.3 = 70 units
```

### Special Cases

| Scenario | Handling |
|----------|----------|
| No velocity data (new item, never sold) | Don't include in PO. Flag as "no_data" — requires manual decision. |
| Out of stock, has velocity | Include with suggested qty. Mark as "critical." |
| Very low velocity (< 0.01 units/day) | Include but flag as "slow mover." Suggest minimum practical order qty (e.g., 1 unit). |
| Velocity only from online (no wholesale) | Include. The item still needs stock regardless of channel. |
| Current stock covers lead time | Status is "OK" — not included in PO by default. User can toggle to include. |

## PO Generation Workflow

### Step 1: Select Brand
User clicks "Build PO" from Brand Overview, or selects brand from dropdown.

### Step 2: System Pre-fills PO
System queries all SKUs for that brand where `reorder_status IN ('critical', 'warning', 'out_of_stock')`. For each:
- Calculates suggested order quantity using default lead time and 1.3x buffer
- Pre-fills the editable "Order Qty" column with the suggested amount

### Step 3: User Adjusts Settings
User can change:
- **Lead time** (sea/air/custom) → recalculates all suggested quantities
- **Safety buffer** (1.0x to 2.0x slider) → recalculates all suggested quantities
- **Include "warning" items** toggle → adds/removes warning-status SKUs
- **Include "OK" items below threshold** → adds SKUs that are OK but approaching warning

### Step 4: User Edits Individual Items
For each row, user can:
- Uncheck the "Include" checkbox to exclude from PO
- Edit the "Order Qty" to any number
- Add notes (e.g., "rush — needed for exhibition")

### Step 5: Export
User clicks Export. System generates:

## Linking Suppliers to Brands

One supplier can supply multiple brands (stock categories). The link is:

```
suppliers.tally_party → parties.tally_name (where channel = 'supplier')
```

And within transactions, the supplier party appears in Purchase vouchers for stock items that belong to specific categories. The system can auto-detect this link:

```python
def detect_supplier_brand_mapping(db_conn) -> dict:
    """
    Find which supplier supplies which brands by looking at
    Purchase transactions: supplier party → stock item → category.
    """
    cursor = db_conn.cursor()
    cursor.execute("""
        SELECT DISTINCT 
            t.party_name as supplier,
            si.category_name as brand
        FROM transactions t
        JOIN stock_items si ON si.tally_name = t.stock_item_name
        WHERE t.channel = 'supplier' AND t.is_inward = TRUE
        ORDER BY supplier, brand
    """)
    
    mapping = {}
    for supplier, brand in cursor.fetchall():
        if supplier not in mapping:
            mapping[supplier] = []
        mapping[supplier].append(brand)
    
    return mapping
```
